            ==============================================================================
                                    WHat is a Microcontroller
            ==============================================================================
So, a MCU is an integrated circuit(IC) device used for controlling other PORTIONS of an electronic system, usually via a 
a microprocessor unit(MPU), memory and some peripherals[all of which are typically packaged into one MCU/MCU board].

A microcontroller consists of a central processing unit(CPU), nonvolatile memory, voltaile memory, peripherals and support 
circuitry.

                                @@Central Processing Unit@@
The CPU performs arithmetic operations, manages data flow, and generates control signals in accordance with the sequence of instructions
created by the programmer. 

It is a bit of electronic circuitry that has a bunch of predefined logical and mathematical operations built in, and knows where to find
a list of these operations to follow(your program) and where to get the data it needs to execute them
                                        @@Memory@@
Nonvolatile memory is used to store the MCU program which is a list of machine-language instructions that tell the CPU exactly what to do.
Typically known as "flash" storage

Volatile memory(RAM) is used for temporary data storage. This data is lost when MCU loses power. Internal registers also provide temporary
data storage, but we don't think of these as a seperate functional block(non-volatile vs volatile) because they are integratged into the
CPU.

                                    @@Peripherals@@
We use the word "peripheral" to describe the hardware modules that help a MCU to interact with the external system. This includes but not 
limited to....
    --Data Converters: analog-to-digital converter, digital-to-analog converter, reference-voltage converter
    --Clock Generation: internal oscillator, crystal-drive circuitry, phase-locked Loop
    --Timing/Timers: general-purpose timer, real-time clock, external-event counter, pulse-with modulation
    --ANalog/Signal Processing: operational amplifier, analog comparator
    --Input/Output: general-purpose digital input and output circuitry, parallel memory interface
    --Serial Communication: UART,SPI,I2C, USB, CAN
                                @@Support Circuitry@@
Microcontrollers incorporate a variety of functional blocks that cannot be classified as peripherals because their primary purpose, is NOT to
control, monitor or communicate with external components. But they are still important....
    --Debug Circuitry: allows the designer to carefully monitor the MCU as it is execueting instructions. Method of tracking down bugs and
    optimizing firmware performance.
    
    --Interrupts: is a software function that you can write that automatically executes whenever an interrupt condition is met. They're called
    interrupt service routines because the processor stops whatever it was doing in the main flow of your program and runs the appropriate function.
    After it is done with the interruput routine, the processor picks up your program's normal operation.
    These are an extremely valuable aspect of MCU functionality. Interrupts are generated by external or internal hardware-based events,
    and they cause the processor to immediately respond to these events by execueting a specific group of instructions.
    
    --Clock-generation module: can be considerded a peripheral if it's intended to produce signals that will be used outside the chip, but in many
    cases the primary purpose of a MCU internal osciallator is to provide a clock signal for the CPU and peripherals.
    Internal osciallators often have low precesion, but in applications that can tolerate this low precesion, they are convenient and effective way
    to simplify the design adn save board space.
    
    --Power-Supply Circuitry: integrated voltage regulators allow for on-chip generation of required supply voltages, power-management modules can
    be used to significantly reduce the device's current consumption during inactive states, and supervisor modules can place the processor in
    a stable reset state when the supply voltage is not high enough to ensure reliable operation.(i think these might be the different power modes
    the MCU can be in depending on the code that you write.)
            ==============================================================================
                                    Microcontroller Datasheets
            ==============================================================================
Sometimes MCU datasheets(or just any datasheet in general) present an overwhelming abundance of facts, figures, and specs. So how do you guide
yourself to the information necessary to conduct your projects.

As microcontrollers become increasingly sophisticated and powerful, their datasheets become longer and complex. Despite being very informal about
the specifications of the device it can present some issues to the reader....

It can be an insane obstacle to overcome for students and engineers who do not have experience with MCU development. For basic applications that rely
on example code and library functions, it may be possible to get the job done without even looking at the datasheet. In most cases though, it is essential
to consult and even study a MCU datasheet.
So, a guideline can be set to aim for those spec.

    1.Assess the Nature of the Documents: Various types of inormation may be distributed among MULTIPLE documents, and you would need to breifly examine these
    documents in order to determine which ones have the specs, descriptions, and recommendations that you really need during a particular phase of the 
    development process. These can be...
        **Datasheets: Focuses on hardware. Tells about the physical dimensions, pinouts, electrical limits(voltage/current), and timing characteristics
        **Reference Manuals(RM): Focuses on architecture and software. For an MCU, this is where you find the memory map, how the peripherals(Timers,ADC, SPI)
        work, and the specific register addresses you need to manipulate in C/C++
        **Application Note: Highly specific guides written by the manufactureer's engineers showing you how to use the component for a specific task(e.g, 
        "Designing a PCB antenna for 2.4GHz" or "Motor control using PWM")
        **The Errata:Lists all hardware bugs and silicon flaws found after being shipped, along with software workarounds.
    
    2.Ignoring the Core: Since most people will probably write MCU code in C/C++, there typically is no need to go in depth of to the entire specs of the CPU
    (depending on scenario but ideally most of the time this only concerns the compiler).
    The bottom line is to think about all of the processor-related details you DO NOT need to woory about when you're writing code in C/C++, and then try to 
    identify the corresponding sections of the datasheet so that you can skip them.
    
    3.Do no Scrutinize the Electrical specifications: Semiconductor manufacturers typically do an excellemt job of thoroughly characterizing their devices.
    That's awesome!.....but you typically only need a small proportion of a device's electrical specifications related to your project.
    If there are a few specs that are particularly critical for your application, by all means check them, but just remember that empirical data gathered BY
    YOUR system under your operational parameters will be more valuable than the numbers in the datahseet.
